Chapter 7 · Optimisation

Toolpath Ordering

After every chain source has run (Ch. 3 - Ch. 6), you hold an unordered set of chains. Drawn in arbitrary order, the tool wastes most of its time on pointless travel between distant chains. Ordering finds a good visiting sequence and the best orientation for each chain - purely as geometry, with no dialect involved.

1. The problem

Given chains C₁…Cₙ, each with two ends (or, for a closed loop, any vertex can be the starting point), and a start position, we need to choose:

  1. a permutation (visiting order), and
  2. for each chain, a direction (which end to enter from; where to start a loop),

that minimises the total pen-up travel (the distance between consecutive chains).

Travelling Salesman Problem (TSP)

This is a variant of the Travelling Salesman Problem (TSP), with one twist: each "city" is not a point but an edge/loop with multiple possible entry points. Exact TSP is NP-hard, so we use a fast greedy heuristic - the travel savings from any reasonable ordering already dwarf the gap between the heuristic and the true optimum.

2. Greedy nearest-neighbour (the workhorse)

The workhorse algorithm, simple and cheap, is the greedy nearest-neighbour heuristic: always move to the unplaced chain whose end is nearest to the current position.

cur = startPoint
while unplaced chains remain:
    // pick the unplaced chain whose NEAREST end to "cur" is closest
    pick chain c minimising min(|cur − c.first|, |cur − c.last|)
    orient c to start at its nearer end   // see §3
    append c to the result; cur = c.last   // the new end after passing through c
  • Complexity O(n²) (each step scans all remaining chains). Fine up to a few thousand chains.
  • Uses squared distance for comparisons (no sqrt needed).
  • Deterministic and good enough in practice: nearest-neighbour usually lands within a small constant factor of the optimum (the theoretical worst case is much looser, but real toolpaths are not adversarial) - moreover the fill sources already emit chains in a locally-coherent order (scanline / ring order), so the greedy pass is mostly just stitching and flipping them.
Keyword

greedy nearest-neighbour TSP heuristic

3. Chain orientation & loop re-rooting

Choosing which end to enter from matters just as much as the visiting order:

  • Open chain: reverse it if and only if its last point is nearer to cur than its first point. (Only two choices; pick the nearer entry.)
  • Closed loop: a loop can start at any vertex. Re-root it to begin (and end) at the vertex nearest cur, which nearly eliminates the travel needed to reach that loop:
// re-root a closed loop to the vertex nearest "cur"
drop the duplicate closing vertex (last point = first point)
find the index of the vertex nearest "cur"
rotate the ring so that vertex is first
re-append the first vertex to re-close the loop
A big win, at almost no cost

Re-rooting closed loops is a large improvement for near-zero cost. Without it, every loop is entered at whatever vertex its author happened to set as the starting point - which could be on the far side from cur.

4. Scaling up: when n is too large

Dense fills (fine line spacing over a large area) can produce tens of thousands of tiny chains. Beyond a cutoff (e.g. around ~4000 chains), the O(n²) pass becomes too slow. There are two safe fallbacks:

MethodComplexityWhen to use
Greedy nearest-neighbourO(n²)Below the cutoff (~4000 chains) - the best quality for an acceptable cost.
Orientation-onlyO(n)Above the cutoff, when the source already emits in a coherent scanline/ring order - keep the source order, only flip/re-root each chain relative to the running position.
Spatial bucketing (grid/Hilbert)O(n log n)Above the cutoff, when the source is not already locally sorted - sort chain midpoints by grid or Hilbert curve, then greedily order within each bucket.

The first option (trusting the source's locality) leverages the fact that fill sources already emit chains in a scanline / ring order that is nearly travel-optimal. Skip the reordering step and apply only per-chain orientation (still flipping/re-rooting relative to the running position); this preserves boustrophedon ("ox-plough" back-and-forth) locality at O(n) cost.

Always set a cap

Always cap the expensive pass; never let ordering itself become the bottleneck of the whole pipeline. This ties directly into the system's overall performance budget - see also Ch. 11 · Performance & Complexity.

5. Per-group ordering

Order within a coherent group, not across the whole document, because groups (layers/tools) carry different machine settings and must be emitted contiguously:

for each layer/tool-group in output order:
    collect that group's chains
    ordered = orderChains(chains, currentPosition)
    emit ordered   // carrying this group's machine settings
    currentPosition = last emitted point   // carried into the next group

Carrying currentPosition across groups means the first chain of the next group is chosen relative to where the previous group just ended - with no wasted return-to-origin trip.

Optional: intra-group affinity - keeping compound paths together

Greedy ordering treats every chain as an independent "city", which can scatter the subpaths of one compound path or one glyph (outer boundary + its holes) across the whole tour. If you need them to stay contiguous (e.g. cut a glyph's counter immediately after its outline), tag chains with an optional affinity key, then order groups of chains sharing a key as a unit: order within the affinity group first, then treat the whole group as a single "super-city" in the outer nearest-neighbour pass. Skip this technique if raw travel-minimisation is the only goal.

Heterogeneous sources within one layer

A layer often mixes a shape's fill chains, its outline chains, and nearby single-stroke text. Pure nearest-neighbour will happily interleave them (drawing text in between a fill region and its outline). For a job at a single power level, this only affects travel and is therefore harmless - but when order affects quality (fill before outline so the edge stays crisp; see Ch. 4 §2b), give the fill and outline of the same shape a shared affinity key and emit fill-then-outline within that group. Ordering by itself gives no such guarantee - the affinity key is the mechanism that enforces it.

6. Optional refinement (2-opt)

If output quality matters more than generation speed, run a few extra 2-opt passes over the greedy tour: repeatedly reverse a sub-sequence of the tour if doing so shortens the total travel. Cheap, and it removes the obvious "self-crossing" moves that the greedy algorithm leaves behind. Rarely needed for drawing/engraving, but a standard technique when you want a tighter tour.

7. What ordering must NOT do

Boundaries of responsibility
  • It must not merge or split chains (that changes the artwork).
  • It must not emit any command (that is the emitter's job).
  • It must not reorder across a group boundary (that breaks per-group machine settings).
  • It must not change a chain's point sequence except by reversal / loop re-rooting (an angle-preserving relabelling).

In other words, ordering is a purely geometric stage: the input is a set of chains, the output is the same set of chains, only their order and (possibly) their direction differ. The next stage - code emission (Ch. 8) - is the only place that knows about a specific machine dialect.

8. Checklist (for AI generating the ordering stage)

Checklist
  • Greedy nearest-neighbour on minimum endpoint distance, using squared distances.
  • Orient open chains by their nearer end; re-root closed loops to the nearest vertex.
  • Cap the O(n²) pass; above the cap, fall back to orientation-only (O(n)).
  • Order per layer/tool-group; carry the running position across groups.
  • Keep it pure geometry: the output is an ordered chain list, nothing is emitted.

With a well-ordered set of chains in hand, all that remains is to translate them - along with the modal tool state and coordinate formatting - into a concrete machine language. That is the subject of the next chapter. For pathological geometric cases (empty chains, single points, duplicates) that can affect both ordering and emission, see Ch. 12.
