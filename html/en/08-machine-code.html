Chapter 8 · Emission

Machine-Code Emission

This is the only stage that knows about a specific machine language. It takes the ordered chain list (Chapter 7) and serialises it into a concrete command language - GRBL G-code, HPGL, Marlin, and so on. Everything upstream is entirely machine-neutral; everything machine-specific lives here, and only here.

1. The emitter interface

Model the emitter as a small set of operations. Each dialect (concrete machine language) implements every operation; the driver loop calls them uniformly, with no idea which machine it is talking to:

interface Emitter:
    preamble()                       # units, absolute mode, tool init, return home
    travelTo(point)                  # move while the tool is DISENGAGED (rapid)
    engage([power])                  # pen down / laser on / blade down
    workTo(point[, power])           # move while the tool is ENGAGED
    disengage()                      # pen up / laser off / blade up
    setFeed(rate)                    # working speed (if the dialect has it)
    comment(text)                    # human annotation (if the dialect supports it)
    postamble()                      # disengage tool, return home, end
    format(point) -> string          # world→machine transform + unit scaling

The driver loop is identical for every dialect:

preamble()
cur = home
for each group:
    setGroupParams(power, feed, force, passes)
    for pass in 1..passes:
        for chain in orderedChains(group):
            if dist(cur, chain.first) > ε:
                disengage(); travelTo(chain.first)
            engage(power)
            for p in chain[1:]: workTo(p[, power])
            cur = chain.last
postamble()

Only preamble/travelTo/engage/workTo/disengage/format differ between dialects. Everything else - modal state, group ordering, pass count - only needs to be written once.

2. The dialect mapping table

For three representative dialects, here is how each abstract operation translates to a concrete command:

OperationGRBL (G-code)Basic HPGLMarlin-ish
units / initG21 G90IN;G21 G90

work (engaged)G1 X.. Y.. [F..]PD x,y;G1 X.. Y..

HPGL has no distinct "engage" command

In HPGL, pen-down is the working move: PD x,y; lowers the pen (if raised) and draws to x,y; PU x,y; raises the pen and travels there. So the abstract engage + workTo collapse into a single PD command - and, symmetrically, disengage + travelTo collapse into a single PU command. In the driver loop, model HPGL's engage/disengage as state-only no-ops (they emit nothing); the actual pen up/down rides the next PU/PD coordinate command. The modal bool tracker still produces the correct PU(travel) → PD(work) sequence.

GRBL laser mode

Enable $32=1 so the beam only fires during motion. Set power S on the M3/M4 that begins a working run (it takes effect on the following move); never leave the laser on across a G0 travel. G0 travels use the controller's rapid rate - do not attach a working F to them (feed applies only to G1).

GRBL's S range is 0 .. $30, where $30 is the controller's configured max-spindle-speed setting (the value corresponding to 100% power - commonly 1000). This is not a fixed ceiling of 30. The formula from percent power to S is: S = round(power% / 100 × $30).

GRBL engage mode: use M3 (constant power) for vector working moves, and M4 (dynamic power, beam scales with actual speed) for raster (see Chapter 9 §5).

This abstraction holds because all these machines share the same engage/travel/work/disengage model (see Chapter 0 §1).

3. Modal state tracking (never emit redundant commands)

Most controllers are modal: a setting persists until it is changed. Track it and emit a command only when something actually changes:

  • Engage state. Keep an engaged bool. Only emit engage when going disengaged→engaged, only emit disengage when going engaged→disengaged. Redundant toggles waste time and can dwell the tool.
  • Power (S). Track lastPower; omit the S word on a working move that keeps the same power. (Essential for raster, Chapter 9, where power changes per scanline.)
  • Feed (F). Emit once per group (it is modal); re-emit when a group's feed differs from the previous group's.
  • Coordinates. Some dialects allow omitting an unchanged axis; usually not worth the added complexity - just emit both X and Y.
Why this matters, beyond aesthetics

Shorter lines pack more moves into the controller's receive buffer, giving the planner deeper look-ahead and smoother motion on dense paths (see Chapter 10 and Chapter 11). Trimming coordinate precision helps too (§4).

4. Coordinate formatting

  • Precision: emit at most ~3 decimals (micrometre scale) and strip trailing zeros (12.500 → 12.5, 7.000 → 7). Parsers read the short form identically, and shorter lines stream better. Normalise -0 → 0.
  • Unit scaling: apply the machine's unit. GRBL/Marlin use mm directly; basic HPGL uses plotter units = 1016/25.4 ≈ 40 units/mm (1 unit = 1/1016 inch) and integer coordinates, so round after scaling.
  • Do this once, at the end. format() applies the world→machine transform (§5) and the unit scale to each point, exactly at emission time - never earlier.

5. World → machine coordinate transform

Design space (world) and machine space rarely coincide. Compose a small affine transform, applied inside format():

  • Home corner / axis direction. Translate + flip so the origin is the machine's home and the axes point the way the hardware expects. (Example: design +Y points down, machine +Y points up ⇒ y' = boardH − y.)
  • Mirroring. Some devices home at one side with an inherent reflection (e.g. a cutter homing at the right with +X running left). Pre-mirror in the emitter to cancel it out (x' = boardW − x) so the output isn't reversed. A separate, user-facing "mirror" toggle (for iron-on/HTV) is applied before this step, affecting only the emitted text - not the preview.
  • Rotation. To fit portrait artwork onto a landscape bed, or vice versa, rotate the whole job (e.g. 90° clockwise: (x,y) → (y, −x), then re-anchor to the positive quadrant).
  • Rotary wrapping. If one axis is a rotary attachment, convert that axis' linear length (mm) into degrees: deg = mm / mmPerRev × 360, and emit it on the rotary axis letter (e.g. replace Y with A), leaving the remaining linear axis unchanged.
Keep home moves in raw machine coordinates

Moves that return home must stay in raw machine coordinates (not user-mirrored) so that "home" is always the device's true origin. Preview parity: keep the simulator in world space (readable to the user); only the emitted text carries these transforms (see Chapter 1 §8-9).

6. Preamble, postamble & comments

  • Preamble: set units + absolute positioning mode, initialise the tool (force/speed/power/pen select), and move to home.
  • Postamble: disengage the tool, return home, deselect/end.
  • Comments/annotations: dialects with a comment syntax (G-code ;) can carry human-readable notes (layer names, settings, coarsening warnings). Dialects without comments (basic HPGL) must be sent only bare commands - so keep annotations as a separate, clearly-marked list, and strip them before streaming to the device (e.g. drop any line starting with ; or any blank line). Never send a comment to a machine that can't parse one.

7. Passes & groups

  • Passes: repeat a group's already-ordered chains N times for thick material or a bolder mark. Order once, repeat the result.
  • Groups (layers/tools): emit as contiguous blocks, each carrying its own power/feed/force/interval. Order within a group; keep position continuous across groups (see Chapter 7 §5).

8. Checklist (when an AI generates an emitter)

  • Implement the Emitter interface; keep the driver loop dialect-free.
  • Track engage/power/feed modal state; emit only on change.
  • Format coordinates: trim precision + strip trailing zeros, scale to device units.
  • Apply the world→machine affine (home/mirror/rotate/rotary) only inside format(); keep home moves in raw machine coordinates.
  • Emit preamble + postamble; strip comments for dialects that don't support them.
  • Repeat ordered chains for passes; emit groups contiguously with their settings.
  • Drive the simulator from the same ordered chain list, in world space.
