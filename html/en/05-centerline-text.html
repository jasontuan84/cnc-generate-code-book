Chapter 5 · Text

Single-Stroke (Centerline) Text

Normal vector text is a set of filled glyph outlines: the letter "I" is really a thin rectangle with four sides. A pen or engraving head asked to reproduce it must draw both edges of every stroke (and then fill the gap between them) - slow, and it looks doubled. Single-stroke text instead draws each glyph's skeleton / medial axis - the centerline of every stroke - in a single pass. This is the "engraving font" / "stick font" behaviour.

There is a shortcut: use a real single-line font

If you already have a genuine single-line font (Hershey / a dedicated engraving font), use its centerline strokes directly - no extraction needed. The pipeline below is for the general case where you only have filled outlines (any TrueType/OpenType font, imported vector artwork, or an arbitrary shape). This chapter describes how to derive a clean centerline from any filled glyph outline; it is font- and language-independent.

1. The problem: why single-stroke is needed

Look closely at the letter "I". On a point-to-point machine, a filled "I" is not a single straight stroke: it is a closed loop of four edges, and filling the interior means the machine sweeps back and forth many times. The result is three drawbacks at once: the stroke looks twice as thick as intended, run time multiplies, and at small sizes the two edges merge into a single blobby mass. Yet the geometric meaning of "I" is only a single vertical segment.

Medial axis / skeleton

The medial axis, also called the skeleton, is the set of points equidistant from the two nearest boundaries of a shape. For a stroke of even width, the medial axis is the "thread" running down its middle. Extracting that thread and drawing it once is the whole goal of this chapter.

2. When it applies

Single-stroke mode is activated only when both of the following are true:

  • The job (or the layer) has the "stroke text" option enabled, and
  • the shape is text (or any shape you choose to skeletonise).

When active, it replaces the shape's normal outline/fill chains: instead of tracing the glyph boundary, the system emits the extracted centerlines. This applies regardless of tool type (pen, laser, blade); other shapes are unaffected.

Where this sits in the overall pipeline

This is a chain source at stage 3 (plan chains) of the overall architecture. The input is a flattened glyph outline (Ch. 2); the output is centerline chains in world coordinates (mm), ready for ordering and code emission.

3. Pipeline: raster skeletonisation

Deriving the medial axis analytically straight from Bézier curves is hard and brittle. The robust, universal method is raster-based skeletonisation: rasterise the glyph into a bitmap, thin it down to a one-pixel skeleton, then trace the skeleton into polylines. The ten steps below form the complete pipeline.

1 · rasterise→ 2 · thin→ 3 · prune→ 4 · trace→ 5 · join→ 6 · smooth→ 7 · simplify→ 8 · round→ 9 · map mm→ 10 · extend

4. Step by step

4.1 Rasterise (fill the glyph into a bitmap)

Render the filled glyph path into a grayscale / 1-bit bitmap. Choose a resolution res (pixels per mm) high enough to resolve the thinnest stroke (a few px across) but capped so the bitmap does not balloon (e.g. max dimension around 2600 px, at most about 9 million pixels). Disable anti-aliasing, since we want a crisp ink/no-ink boundary; fill using the non-zero winding rule (so overlapping contours merge into solid ink); threshold to ink = 1 / background = 0. Add a small padding border so edge pixels also have neighbours.

4.2 Thin - Zhang–Suen

The classic Zhang–Suen thinning algorithm iteratively removes boundary pixels that are not essential to connectivity, until only a 1-pixel-wide skeleton remains. Each iteration has two complementary sub-passes with different deletion masks; it uses:

  • B(p) = the number of 8-neighbours that are ink (must lie in the range 2-6 to be deleted),
  • A(p) = the number of 0→1 transitions going around the 8-ring (must be exactly 1),
  • two corner-condition products, which differ between the two sub-passes.

Repeat until no pixel changes (guarded by a max-iteration cap). Keywords: Zhang–Suen thinning / morphological skeletonisation.

4.3 Prune spurs

Thinning spawns short dead-end "twigs" at serifs, stroke terminals, and junctions, making the centerline look hairy. Prune each branch that runs from an endpoint (1 neighbour) into a junction (>2 neighbours) within maxSpur pixels: delete the twig, keep the junction. Set maxSpur a little above a typical stroke half-width (about 0.7 mm converted to pixels) so real strokes (much longer) survive while noise artefacts vanish. Run a few passes.

Do not prune an entire stroke

Never prune a branch that runs between two genuine endpoints: that is a whole stroke (like the letter "l"), not a spur. Only a branch with one endpoint end and one junction end, short enough, is a pruning candidate.

4.4 Trace (skeleton → polylines)

Walk along the 1-pixel skeleton to trace out polylines. Start from every node (endpoint or junction) and follow the runs with exactly 2 neighbours, marking each undirected edge used at most once; then mop up the remaining pure loops (rings with no node, like the letter "O"). Result: many short polylines in pixel space.

4.5 Join at junctions (straightest continuation)

The raw trace is fragmented (every junction breaks the chain). Rejoin fragments meeting at a shared endpoint using straightest continuation: at a junction, extend the chain into whichever fragment's direction best matches the incoming direction (maximum dot product, requiring a turn of < 90°). This turns a scribble of disjoint fragments into a few flowing strokes, giving far fewer pen lifts and smooth runs through junctions.

4.6 Smooth - moving average

The raw skeleton has a ±1 px "staircase" wobble. Apply a low-pass moving average filter (e.g. convolution with the [1,2,1]/4 kernel) a few times. Pin the endpoints of open polylines (don't let them move); for closed loops, wrap around. Keywords: Laplacian smoothing / box smoothing.

4.7 Simplify - Douglas–Peucker

Reduce the point count with the Ramer–Douglas–Peucker algorithm at a small tolerance (a finer tolerance keeps rounded curves more faithfully). It removes collinear / near-collinear vertices without changing the shape.

4.8 Round - Chaikin

Apply Chaikin corner-cutting (1-2 iterations) to turn the simplified polyline into a smooth curve. Each iteration replaces every corner with two points at 1/4 and 3/4 along its edges. Keyword: Chaikin subdivision. See also curve continuity in Ch. 10.

4.9 Map to world space (back to mm)

Convert pixel coordinates back to world mm using the inverse of the rasterise transform (accounting for padding and res).

4.10 Extend terminals

The medial axis stops about half a stroke-width short of a stroke's visual endpoint. This is the "missing stroke tip" problem. The fix: extend every endpoint that is a genuine terminal outward along its tangent until it reaches the glyph boundary:

  • A terminal is an endpoint that no other stroke touches within the share tolerance (about 0.35 mm, see §5). An endpoint that lies within that tolerance of another polyline is a junction: leave it alone (extending would overshoot into the glyph). Closed loops have no terminals.
  • Estimate the tangent from a short back-off along the stroke, then step outward in small increments as long as the point remains inside the filled glyph, up to a maximum extension.
Robustness warning: choose the share tolerance carefully

Keep the share tolerance well below the narrowest gap between distinct strokes. Otherwise, glyphs (or serifs) that touch or overlap will cause a true terminal to be misclassified as a junction (producing a missing tip) or vice versa (producing an overshoot spike). For glyphs that touch themselves, the max-extension cap (§5) bounds how far a mis-stepped extension can run along an unintended interior path.

5. Cleanup thresholds (with rationale)

Every tunable parameter has a reason behind it; understanding the reason is what lets you re-tune it intelligently.

Resolution for small type

res is chosen per glyph so the thinnest stroke has enough pixels. Small type (cap height ≲ 4-5 mm) needs a high res (tens of px/mm), or is better served by a real single-line font (§1). Smoothness measured in mm is preserved by DP + Chaikin regardless of res; what a too-coarse raster loses is fidelity (whether the centerline actually matches the glyph).

6. Why this order

Thin first (topology), then prune (remove noise), then trace (extract), then join (connect), then apply geometric smoothing (appearance), and extend terminals as the very last step (because it needs the finished tangents and the "is this a shared junction?" relationship between completed polylines). Reordering breaks these assumptions: for example, smoothing before pruning would blur the very spurs you want to detect and cut.

Principle: topology first, geometry second, relationships last

Three layers of work must be ordered correctly. The topology layer (thin, prune, trace, join) decides "how many strokes and how they connect"; the geometry layer (smooth, simplify, round) decides "how smooth they look"; the relationship layer (extend terminals) needs every stroke finished before it can tell terminal from junction.

7. Checklist (AI generating centerline text)

  1. Rasterise the filled glyph (non-zero fill, no AA, padded, bounded size).
  2. Zhang–Suen thin down to a 1-px skeleton.
  3. Prune spurs into junctions (a few passes).
  4. Trace nodes → polylines; mop up remaining loops; use each edge once.
  5. Join fragments by straightest continuation (turn < 90°).
  6. Smooth (moving average) → Douglas–Peucker → Chaikin.
  7. Map back to world mm.
  8. Extend true terminals to the boundary; leave junctions alone.
  9. Return the chains in world space; they replace the glyph's outline/fill.
Result: elegant text in a single stroke

Done correctly through all ten steps, any filled glyph becomes a handful of smoothly flowing centerline chains, each stroke traced once, tips landing right at the boundary. The output feeds into the same downstream flow as every other source: minimise travel in Ch. 7, then emit code in Ch. 8. The time cost drops sharply because the fill scan lines disappear entirely, exactly as the cost model in Ch. 11 predicts.
