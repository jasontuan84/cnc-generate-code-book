Chapter 2 · Geometry

Geometry Foundations

Every chain source - fill, outline, text, pattern - is built on the same small set of geometric primitives. Implement these once, correctly, and the rest becomes straightforward. All operations here are dialect-independent: they are pure 2-D computational geometry.

Where this chapter sits in the pipeline

This is the toolkit for stage 1 (resolve geometry) and stage 2 (extract primitives) introduced in Chapter 1. After this chapter, every drawing has become clean polylines plus inside/outside information - ready for the fill engines in Chapter 3 and the outlines in Chapter 4.

1. Curve flattening (Bézier / arc → polyline)

Machines move point-to-point, so every curve must be approximated by short straight segments. The only parameter is tolerance: the maximum distance allowed between the true curve and the polyline chord.

The scale-up trick (uniform world-space tolerance)

Many vector libraries flatten with a tolerance fixed in path units, which is too coarse at millimetre scale and produces visibly faceted curves (the flat sides of an "O" or "S"). The robust, library-agnostic fix:

1 · Scale the path up by a factor k (e.g. k = 25) → 2 · Flatten in the enlarged space → 3 · Scale every resulting point back down by 1/k

Keyword to remember: adaptive / subdivision flattening with tolerance control.

Trade-off: smaller tolerance, larger output

Smaller tolerance ⇒ more points ⇒ larger output and (on some controllers) slower streaming. Pick tolerance at "just below the tool's mark width". ~0.02mm is a safe universal default for mm-scale artwork.

Fine vs coarse contexts

Use the fine tolerance for any geometry whose points become emitted output or must align with other output: outlines, glyph curves, and any region boundary whose intersections produce visible fill/pattern endpoints. A coarser tolerance is acceptable only for boundaries used in decisions that never emit a vertex and tolerate ~sub-0.05mm error: shape-metric estimation, tile-count bounding, coarse inside/outside tests.

Never clip a visible fill line against a coarse boundary

Its endpoints would land off the true edge and misalign with a fine outline (this is exactly the seam bug described in 10 §2.4). Centralise both tolerances in one place so the whole system stays consistent.

2. Polyline / polygon representation

  • A polyline is an ordered point list. Open = distinct endpoints; closed = first point ≈ last point (within an epsilon).
  • Test closed by |first − last| < ε and at least 3 (often ≥4) points.
  • A shape with holes is a set of subpaths (outer boundary + inner boundaries). Keep them as separate polylines and let the fill rule decide inside/outside; do not try to merge them into one ring.
Epsilon: compare using squared distance

Use a squared-distance epsilon (ε²) for closeness tests to avoid sqrt. A typical value is ε = 1e-3 mm.

3. Fill rules: even-odd vs non-zero winding

To decide which points are "inside" a multi-subpath shape:

Even-odd (parity)

A point is inside if a ray cast from it crosses the boundary an odd number of times. Direction-independent - subpath winding does not matter. This is the safe default for arbitrary imported artwork, glyph counters, and concentric rings.

Non-zero winding

Sum the signed crossings; inside if the sum ≠ 0. Needs consistent winding direction; can fill regions that even-odd would treat as holes.

The precise decision

The hole/nesting model in §4 is an even-odd model, and even-odd is the safe default for nested holes (counters, donuts) regardless of winding. But even-odd carves an unintended hole wherever two contours overlap (rather than nest) - for example two overlapping same-direction blobs, or an accent overlapping a glyph, that the designer meant to read as solid. So the precise rule is:

  • Subpaths that are nested (a hole fully inside another) → even-odd is correct and winding-independent; use it.
  • Subpaths that overlap and should union into a solid → honor the source's declared non-zero winding fill rule (most fonts and SVGs use it), or pre-union the overlapping same-winding contours (12 §3.2) before the even-odd fill.

This is why glyph rasterisation (05 §3.1) fills with non-zero winding: a font's overlapping contours must read solid. The nesting-depth test (§4) is winding-independent, so it stays correct under either rule.

Point-in-polygon test

The standard ray-casting / crossing-number algorithm: cast a horizontal ray and count edge crossings, using the half-open vertical span rule (a.y > p.y) != (b.y > p.y) to count a shared vertex exactly once. O(n) per query. Used for hole detection and for the "is this midpoint inside?" test when clipping (§6).

// Ray-casting: is point p inside polyline poly?
function pointInPolygon(p, poly) {
  let inside = false;
  for (let i = 0, j = poly.length - 1; i < poly.length; j = i++) {
    const a = poly[i], b = poly[j];
    // half-open rule on the vertical axis: count each shared vertex exactly once
    if ((a.y > p.y) !== (b.y > p.y)) {
      const xCross = a.x + (p.y - a.y) / (b.y - a.y) * (b.x - a.x);
      if (p.x < xCross) inside = !inside;
    }
  }
  return inside;
}

4. Hole & nesting detection (even-odd nesting depth)

To know which subpaths are solid and which are holes:

  1. For each subpath, compute its signed area using the shoelace formula: A = ½ Σ (xᵢ·yᵢ₊₁ − xᵢ₊₁·yᵢ). |A| is the area; the sign gives the winding.
  2. Compute a nesting depth for each subpath by testing one of its vertices against every larger subpath with point-in-polygon. Depth = number of larger subpaths that contain it.
  3. Even depth ⇒ solid (adds to the net area); odd depth ⇒ hole (subtracts).
Why test a vertex, not a centroid

Two concentric rings (a donut, or the outer/inner of a letter "O") can share the same centroid, so a centroid test fails. A boundary vertex is unambiguously inside or outside the other ring. This makes nesting winding-independent and correct for both donuts and letter counters.

Net area = Σ(solid areas) − Σ(hole areas). Hole count = number of odd-depth subpaths.

5. Convex hull, oriented bounding box & shape metrics

The automatic fill selector (Chapter 3) classifies a region from cheap scalar metrics. Compute them once:

Convex hull

The Andrew's monotone-chain algorithm: sort points by (x, then y), build the lower then upper hull with a cross-product turn test, O(n log n). The hull gives both a convexity measure and the basis for the oriented bounding box.

Minimum-area oriented bounding box (OBB)

Rotating calipers, based on the theorem that a side of the minimum-area rectangle must be collinear with a hull edge: for each hull edge, project all hull vertices onto that edge's direction and its perpendicular, take the extents, compute the rectangle's area, and keep the minimum. Record:

  • L = long side length, W = short side length,
  • θ = orientation of the long axis (degrees),
  • center.

These metrics are the entire input to the auto-fill decision - no ML, no heuristic soup, just interpretable geometry. The identity mean stroke width = 2A/P is the "average thickness" for a long thin region (area ≈ length × width, perimeter ≈ 2 × length).

6. Clipping a polyline to a region

Fills and patterns generate lines across the whole bounding box, then keep only the parts inside the region. Two common needs:

Straight scanline against subpaths (fast path)

For a horizontal scanline at height y, find every edge crossing, sort the x values, and pair them: [x0,x1] [x2,x3] …. Each pair is an inside span (this is exactly the even-odd rule applied to a single line). Used by hatch / zigzag (Chapter 3); O(edges) per scanline.

Arbitrary polyline against an arbitrary region (general path)

For each segment of the polyline:

  1. Find all intersection parameters t ∈ (0,1) with the region boundary.
  2. Sort them; they split the segment into sub-segments.
  3. Keep a sub-segment if its midpoint lies inside the region. Any even-odd inside-test works here - the hand-rolled ray-cast from §3, or a geometry library's own even-odd contains().
  4. Concatenate adjacent kept sub-segments into output chains; a gap flushes the current chain.

Keyword: segment–polygon clipping. Used by space-filling curves and pattern tiling.

Half-plane clipping (for cells)

Sutherland–Hodgman polygon clipping against a half-plane is the primitive for Voronoi cells (Chapter 3): clip a polygon to the side of a bisector line closer to one site.

7. Offsetting a region inward (inset)

Contour / spiral fills and kerf compensation need an inward offset of a closed region by a distance d. Repeatedly insetting by the line spacing yields the concentric rings those fills draw. There are two robust constructions; pick according to what your environment provides.

Method A - stroke-and-subtract (boolean erosion)

A library-agnostic construction using a stroker:

  1. Stroke the boundary with pen width 2d (the stroke straddles the outline, d inside and d outside), using miter joins and flat caps.
  2. Subtract that stroke band from the original region.
  3. The result is the region shrunk by d on every side; holes grow inward correctly, and thin necks automatically pinch off into separate pieces.

Keyword: polygon offsetting / erosion (a.k.a. Minkowski erosion by a disc; stroke-and-subtract approximates it with the chosen join style). This is exact and resolution-independent, but it requires a robust boolean/offset engine (a Clipper-class library).

Do not substitute a naive per-vertex inset

Do not use the naive inset that "moves each vertex a distance d along its angle-bisector normal": it self-intersects the moment the region is genuinely concave (a star point, the inside of an "L"), and it has no notion of holes or of a region splitting into several pieces - it silently produces crossing, garbage rings. If a per-vertex inset is all you have, it is safe only for a single, near-convex ring; detect anything else and fall through to Method B.

Method B - distance field + iso-contours (topology-robust, dependency-free)

When no boolean-offset library is available - or the artwork routinely has holes, concavity, or many disjoint regions - offset without any boolean operations by reading the inset off a distance field. This is the standard basis for CAM "contour-parallel" / "peel" machining, and it is completely topology-agnostic.

1. RASTERISE the even-odd interior (§3, §6) into an inside/outside mask on a grid
   with step  h ≈ spacing / 3.  (h a fraction of the line spacing keeps the iso-
   lines smooth rather than staircased; pad the grid by a couple of cells of
   guaranteed-outside border so boundary cells resolve. Cap total cells and
   coarsen h if the region is large - see 11 §3.)
2. DISTANCE TRANSFORM: compute, for every interior cell, the distance to the
   nearest exterior cell. A two-pass CHAMFER transform with weights (1, √2)·h
   differs from true Euclidean by at most ~8% (it always over-estimates, worst
   near the ~22.5° diagonal) - negligible against the line spacing - and needs
   no libraries. (An exact separable Euclidean DT - Felzenszwalb–Huttenlocher -
   or a library EDT is a drop-in upgrade when available.)
3. ISO-CONTOURS: the inward offset by distance d is exactly the iso-line
   { distance = d }. Extract iso-lines at  d = ½·spacing, 1½·spacing, 2½·spacing …
   with MARCHING SQUARES (linear interpolation of the crossing point on each
   cell edge), LINK the resulting segments into closed rings by matching
   quantised endpoints, and apply one CHAIKIN pass to shave the grid staircase.
Why it is robust

Holes, concavities, thin necks, and several disjoint blobs are all just features of one scalar field - the iso-lines split, merge, round and vanish correctly with no special cases and no self-intersection. The first ring at d = ½·spacing sits half a line-spacing inside the boundary (the standard contour start point, 03 §3.4). Each level can yield several rings (one per nest); return them all, closed (first point == last point).

Cost: the transform is O(cells); marching squares is a single pass, O(cells + total contour length). Both are bounded by the cell cap. See 11 §3.

Method A vs Method B trade-off

Offsetting via a distance field is an approximation at grid resolution (corners read slightly rounded when h is coarse), but it never produces invalid geometry and needs no dependencies. Method A is exact but needs a real offset engine. A good default: Method B as the always-correct primitive, with Method A (or a library offset) as an exact fast path for the simple single-ring case.

Keywords: distance transform, chamfer distance, marching squares, iso-contour, contour-parallel machining.

Degenerate case (either method)

If the region is thinner than d on every side, the inset is empty - detect this and stop (or keep the last non-empty ring). With Method B this is simply "no cell has distance ≥ d".

8. Numerical hygiene

  • Use squared distances for comparisons; only take sqrt when a real length is needed.
  • Offset scanlines by half a step (y0 = ⌊top/spacing⌋·spacing + spacing/2) so a line never lands exactly on a vertex (which would double-count crossings).
  • Nudge "inside?" probe points a hair (≈0.25·spacing) toward the region's bounding-box center (a cheap approximate centroid) when testing points that lie on a boundary (connectors that run along an edge), so a valid edge is not rejected by a strict contains() that treats the boundary as outside.
  • Scrub every flattened polyline before use: drop non-finite coordinates (NaN / ±inf from a malformed transform or a degenerate curve) and collapse consecutive duplicate points (within a tiny ε). A stray NaN would otherwise reach the emitter as X nan; a zero-length segment causes a divide-by-zero in every downstream tangent/normal/offset computation. Do this once, at extraction time, so no later stage has to defend against it.
  • Cap iterative constructions (rings, curve order, site count, grid cell count) so pathological input (tiny spacing, huge area) always terminates.
Wrapping up this chapter

You now have a complete set of primitives: tolerance-controlled flattening, polyline representation, two fill rules with their point-in-polygon test, nesting depth to identify holes, convex hull + OBB + shape metrics, polyline clipping, and two ways to offset inward. Chapter 3 uses these metrics directly to auto-select a fill strategy; Chapter 4 uses offsetting to build contours and kerf compensation.
