Chapter 1 · Architecture

Architecture & Pipeline

Chapter 0 told us the destination: every drawing reduces to a set of chains of points. This chapter answers the next question: what system produces that set of chains in an organised way, so that adding a new machine or a new fill algorithm doesn't bring the whole building down? The answer is a strictly layered architecture, with five sequential stages and four boundary contracts that must never be broken.

1. The layered responsibility model

Picture the whole system as a pipeline of four functional blocks, running from WORLD space (the design, in the mm units the user sees) to DEVICE space (the machine, in the units its controller understands). Each block knows exactly what it needs to know, and nothing more:

The most important feature of this model is that it is unidirectional: information only flows left to right, and no block ever "looks back" at an earlier one for more detail. Geometry doesn't need to know which layer a shape sits on; Toolpath doesn't need to know whether the machine is GRBL or HPGL; Emitter doesn't need to know the fill rule, because by the time it receives data, every geometric decision has already been finalised.

The layer-correctness test

If you cannot write a unit test for a stage without instantiating a specific machine dialect, the architectural layering has leaked. A fill-pattern function that reads machine.type == "grbl" somewhere inside it is a clear warning sign.

2. Stage 1 - Resolve geometry

The goal of the first stage is to turn "a shape with a transform and a style" into world-space outlines, together with the knowledge of which regions are interior. Three things must happen, in this order:

1 · Bake transforms

Compose the shape's local transform with the transforms of every ancestor (group/layer), then apply it to the local outline to produce a path in world space. Rotation, scale, and shear must all be baked before flattening, so tolerance is measured in world units, not in a distorted local unit.

2 · Choose a fill rule

even-odd or non-zero winding. Even-odd is the safe default: "a hole is wherever subpaths overlap an odd number of times" - it is independent of subpath winding direction. Details in Ch. 2.

3 · Classify the geometry role

Which of the three roles in §5 below does this shape belong to - this decision drives the entire downstream branch.

The output of this stage is a resolved path in world coordinates, together with its style: does it have a fill? a stroke? which pattern? what shade/level?

3. Stage 2 - Extract primitives

The goal is to convert continuous curves into polylines, and split the geometry into the two things every downstream stage consumes:

Outline geometry
The boundary polylines themselves - the shape's edge/outline.
Interior area
The closed interior region, used by fill and pattern engines.

The key operation here is curve flattening at a controlled tolerance (see Ch. 2 §1). Every downstream stage operates only on polylines, never on parametric curves - so this stage centralises all curve math in one place, instead of scattering it across the system.

4. Stage 3 - Plan chains (the dispatch)

For each resolved shape, the system must decide which chain source(s) apply - purely based on that shape's geometry and style, not on any "machine mode". This is a style-driven dispatch, independent of the machine:

for each shape in draw_order:
    if shape is raster_image:
        → RASTER path (scanline modulation)   # ch. 9; NOT chains
        continue

    if single_stroke_text and shape is text:
        → CENTERLINE chains                     # ch. 5
        continue

    if shape.has_fill:                        # covers the interior
        if fill_is_pattern and tiling_produced_strokes:
            → PATTERN chains (tile + clip)          # ch. 6
        elif fill_is_solid:
            → FILL-ENGINE chains (auto strategy)  # ch. 3
        else:  # built-in hatch look, or texture producing nothing
            → HATCH chains at the pattern's angles    # ch. 3/6

    if shape.has_stroke or not shape.has_fill: # draws the boundary
        → OUTLINE chains                          # ch. 4

This is the whole decision. Note that it is additive, not exclusive: the last two if blocks can both be true at once for the same shape.

Three additive cases to distinguish

A shape with both fill and stroke: emits both fill chains and outline chains - the interior is covered and the border is drawn separately (for the concrete order/settings/shared-edge decisions this leaves open, see Ch. 4 §2b).

A shape with fill, no stroke: emits only fill chains - the design does not call for a border, so none is invented. Note: not because "the fill reaches the boundary" - inset-based fills typically leave a thin unfilled rim, see Ch. 4 §2.

An unfilled shape (an open path, a shape with only a stroke, or one with neither): emits only outline chains - with no interior, the boundary is the entire artwork.

The dispatch order also encodes two overriding exceptions: single-stroke text entirely replaces the normal outline/fill handling of a text shape (Ch. 5); and raster is handled completely separately because it is not chain-based.

5. The three geometry roles

Every shape falls into exactly one of three geometry roles. Finding the best toolpath for each role is the entire job of Stages 3-5:

RoleDetectionChains producedBest-fit strategy
Filledhas a valid fillfill/pattern chains (+ outline if also stroked)fill engine picks the time-optimal covering path from the region's shape (Ch. 3)
Stroked (outline)has a stroke, or has no filloutline chainsflatten boundary → chains; order for minimal travel (Ch. 4, Ch. 7)
Text (single-stroke)text + option enabledcenterline chainsskeleton extraction so each glyph is a single pass (Ch. 5)
No "cut vs draw" branch

This architecture has no separate branch for "cutting machine" versus "drawing machine". The dispatch depends only on what the geometry is (fill / stroke / neither), so the same logic produces the correct chains for any point-to-point machine. Tool-specific refinements a cutting tool may want (overcut, kerf offset) are optional post-processing on the already-chosen chains (Ch. 4 §4), not a separate pipeline.

6. Stage 4 - Order chains

Take the unordered chain list from Stage 3 and produce an order plus a direction for each chain that minimises pen-up travel. Detailed in Ch. 7. This stage is pure geometry: it never emits a machine command.

Ordering is performed per layer / per tool group, because different layers carry different machine settings (power, speed, force, passes) and must be emitted as contiguous blocks, not interleaved.

7. Stage 5 - Emit dialect

Serialise the ordered chains into machine commands. This is the only stage that knows about the dialect. It has four responsibilities:

Preamble (units, absolute mode, tool init, home) → Track modal tool state (engaged/disengaged, power, feed) → Coordinate-system transforms (home corner, mirror, rotation, rotary wrap) → Postamble (disengage tool, return home, end)

Tracking modal state ensures redundant commands are never emitted - if the laser power is already at 800, there's no need to resend a power-800 command for every segment. Full details in Ch. 8.

8. Four coordinate spaces

Bugs tend to cluster at coordinate-space boundaries. So all four spaces must be named explicitly, never conflated:

1 · Local space
A shape's own coordinates, before its transform is applied.
2 · World / design space
After baking every transform; a common mm plane with a defined origin corner and axis directions (commonly +X right, +Y down, like screen space). All chain production happens here.
3 · Machine space
The device's own coordinate system: origin at the machine's home corner, axis directions per the hardware, possibly mirrored or rotated relative to world space.
4 · Device units
Machine space scaled to the controller's integer/decimal unit (e.g. mm for GRBL; 1/40 mm "plotter units" for basic HPGL).
Rule

Produce and order chains entirely in world space. Convert to machine space + device units exactly once, per point, at emission time. Keep travel-to-home moves in raw machine coordinates, so "home" is always the true device origin, never passed through the world transform.

9. Preview / simulator parity

If the product displays a simulated toolpath, drive the simulator from the exact same ordered chain list the emitter consumes, tagging each move as engaged or travel. This guarantees that "what you preview is what the machine will do". Keep the simulator geometry in world space (readable to the user) and let only the emitted text carry the machine transform - never maintain two independent path computations.

10. Robustness invariants

The following three principles apply to every chain-generation stage, regardless of the specific algorithm:

Never emit "nothing" for a non-empty area

Every planning strategy must have a fallback to a trivially-correct method - ugly beats empty.

Skip incompatible sources gracefully

A blade machine silently drops raster images; a pen machine drops power modulation. Don't error - just omit.

Bounded work

Every generator must cap its output (max rings, max sites, max raster cells, an ordering-optimisation cutoff), so that a tiny spacing on a huge area cannot hang the machine or exhaust memory. When a cap forces the result to be coarsened, record that as a human-readable annotation.

Architecture is the frame; the rest is content

This chapter gave you the frame: four blocks, five stages, four coordinate spaces, three robustness invariants. The following chapters fill in each cell of this frame - starting with the geometry foundations (curve flattening, fill rule, offsetting) in Chapter 2.
