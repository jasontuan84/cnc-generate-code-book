Chapter 3 · The central chapter

Fill Algorithms & Automatic Selection

A machine that only knows how to draw straight lines cannot "flood-fill" a region the way graphics software does; it must cover that region with a set of line chains. This chapter specifies every fill strategy, the cost model used to rank them, the automatic selector that picks the time-optimal strategy from a region's shape - plus how fill density encodes shade. Everything here returns chains in world space and is completely independent of the machine dialect.

Where this sits in the pipeline

This is stage 3 - chain planning - of the five-stage pipeline (Ch. 0, Ch. 1). It consumes the geometric measurements (hull, OBB, the six metrics) from Ch. 2 and produces chains for the ordering stage and then emission. It has no idea what the target machine is.

1. The cost model (why one strategy beats another)

For a fixed area A and a given line spacing s, the engaged (pen-down) length is essentially constant for any line fill:

# engaged length is almost independent of strategy
engaged length ≈ A / s

So engaged distance does not distinguish techniques. What differs - and what dominates real machine time - is:

  1. Number of tool lifts (disengage → travel → re-engage). Each lift costs a fixed dwell plus an acceleration penalty.
  2. Travel distance between disjoint segments (pen-up moves produce nothing).
  3. Number of direction reversals (decelerate → accelerate at each turn).
Conclusion

A connected fill that draws the same lines with almost no lifts always beats a disjoint fill on time. This single fact drives the entire automatic selector (§4): it only ever chooses among the connected fills.

ClassLiftsBest for
Connected (Zigzag, Spiral, Contour*)~0 to a fewtime-optimal; these are the Auto choices
Disjoint (Hatch, CrossHatch)one lift per linea specific look, manual only
Decorative (Hilbert, Peano, Voronoi)variestexture/aesthetics, manual only

* Contour draws separate rings (one lift per ring); Spiral connects them into one stroke.

The lift floor is Spiral

Think of the tool as a pen plotter: the number of pen-lifts is the metric to minimise. Zigzag lifts ~0 times on a convex region but fragments (one chain per disjoint span) across holes and concavities. Contour lifts once per ring. Spiral, given a topology-robust offset (§3.4) and a nest-aware connector (§3.5), lifts exactly once per nested region - the theoretical floor, because a separate nest genuinely cannot be reached without a lift. On a donut this is the difference between ~2 lifts (Spiral) and dozens (Contour, one per ring) or many (Zigzag, one per span per scanline). This is why the selector in §4 prefers Spiral wherever the offset engine can produce rings.

2. Density: mapping shade to line spacing

Line spacing controls how dark a filled area reads and how long it takes to complete.

spacing = shadeFactor(level) × lineWidth
  • lineWidth - the tool's mark width (pen nib, beam kerf). Lines closer than this overlap; lines much farther apart leave visible gaps.
  • shadeFactor(level) - a multiplier per fill level / shade. Darker ⇒ smaller factor ⇒ tighter lines ⇒ more coverage. A representative four-level map:
LevelFactorEffect
Black (100%)0.8lines slightly overlap → solid
Dark gray1.5small gaps
Gray2.5medium gaps
Light gray5.0sparse
Separation of concerns - but not numerically independent

Density (spacing) and strategy (path shape) are chosen by separate mechanisms: the user sets shade via density; the selector sets the path shape. They are not, however, fully independent numerically: the selector reasons about the shape relative to the current spacing (via widthInLines and the tiny-area rule, §4), so a large change in shade - hence spacing - can legitimately move a shape into a different strategy. A region many lines wide at a dark shade may be only one line wide at a very light shade, where a single contour genuinely is the right fill. This is intended: the strategy adapts to how many passes actually fit. What is guaranteed is (a) at a fixed shade the strategy is deterministic, and (b) shade never overrides an explicit user strategy choice - only Auto re-evaluates. If you need the path shape frozen across shades, pick an explicit strategy instead of Auto.

For a dense engraving/burn fill (laser), spacing is instead driven by the beam/material interval (e.g. ~0.15 mm), deliberately set far below any pen spacing floor, so the burn fully covers the area.

Deriving the shade level from fill colour (automatic tone)

When the shade is not set by hand, derive it from the region's fill colour: darker colour ⇒ denser lines, lighter ⇒ sparser - so a greyscale drawing engraves as a believable tonal image with no per-region tuning.

  1. Compute the colour's perceptual luminance (0 = black … 1 = white): 
    # R,G,B in the range 0..1
Y = 0.299·R + 0.587·G + 0.114·B
    The weights matter: a pure red and a pure yellow of equal RGB magnitude read as very different darkness, and this luminance ranks them correctly.
  2. Map luminance to a spacing multiplier on the base (darkest) spacing: 
    spacing(region) = baseSpacing · ( 1 + Y · (Kmax − 1) )
    with the base spacing being the value for a fully dark fill (multiplier 1, densest) and Kmax the sparsest multiplier (a representative value is ~5, i.e. a light grey ends up ~5× sparser - matching the shade-factor table above). This is a continuous form of that table: black → 1×, mid-grey → ~3×, light → ~5×.
Where it lives

Apply the multiplier per region, upstream of the selector, so a lighter region re-classifies legitimately (its widthInLines shrinks - a region many lines wide when dark may be only one line wide when light, §2 above). Keep it a toggle: some jobs want uniform density regardless of colour. A near-white or none fill means "not filled" and produces no fill at all (only its outline, if any). Parse the colour with the host's colour model; fall back to a small hex / rgb() / named-colour parser so the mapping still works standalone.

3. The strategies

Each strategy is described as: idea → algorithm → connectivity → when to use.

3.1 Hatch (parallel scanlines) - disjoint

  • Idea: fill with parallel straight lines at angle θ, spacing s.
  • Algorithm:
    1. Rotate the region into a frame where the hatch lines are horizontal (rotate by −θ about the region center).
    2. For each scanline y (starting from ⌊top/s⌋·s + s/2, stepping by s), find all boundary crossings, sort the x's, and pair them into inside spans (even-odd on a line, Ch. 2 §6).
    3. Emit each span as a 2-point chain; rotate back to world space.
  • Connectivity: one chain per span ⇒ many lifts.
  • Use: manual only - when the user wants a literal "hatch look", a specific angle, or an architectural shading convention. Multiple angle sets give cross-hatch/grid.
  • Keyword: scanline polygon fill.

3.2 Cross-Hatch - disjoint

Two Hatch passes at perpendicular angles (θ and θ+90°), or any angle set. Same disjoint cost; a darker/textured look. Manual only.

3.3 Zigzag (boustrophedon) - connected ★ Auto candidate

  • Idea: the same scanlines as Hatch, but consecutive spans are joined end-to-end into one continuous back-and-forth stroke.
  • Algorithm:
    1. Compute per-scanline inside spans (as Hatch).
    2. Walk scanlines top→bottom, alternating the left-to-right / right-to-left ordering of the spans (this is the boustrophedon reversal).
    3. Append the next span to the current chain iff the connector between the previous span's end and this span's start is short (≤ ~2.5·s) and lies inside the region. Otherwise flush the chain and start a new one.
    4. Concavities and holes naturally split the fill into a few chains where a connector would leave the region.
  • Connectivity: near-zero lifts on convex/rectangular shapes.
  • Orientation: align the scanlines to the region's long axis (from the OBB, Ch. 2 §5) so runs are longest and reversals fewest.
Edge-connector caveat

Connectors run along the boundary, where a strict contains() treats the boundary itself as outside. Nudge the probe point a hair (≈0.25·spacing) toward the region's bounding-box center (a cheap approximate centroid) before the inside-test, or convex fills fracture into many strokes. This nudge is valid for convex-ish regions. On a concave or holed region the bbox center may lie outside the material - the nudge then only makes the inside-test more likely to reject a connector, so it errs toward fragmenting the fill, never toward burning across a gap. Because Auto routes concave/holed shapes to Spiral (not Zigzag, §4), Zigzag normally sees only convex-ish regions; if you force Zigzag on a holed shape, replace the bbox-center nudge with a local inward test (sample several points along the connector, or offset along the segment's inward normal).

  • Use: open convex / rectangular / thin straight regions. The general-purpose, time-optimal fill.

3.4 Contour (concentric offsets) - connected, a few lifts ★ Auto candidate

  • Idea: fill by repeatedly insetting the boundary; each inset is a closed ring.
  • Algorithm:
    1. Start half a spacing inside the boundary (so rings sit centred and do not retrace the exact outline).
    2. Emit the current region's boundary polylines as closed rings.
    3. Inset by s (Ch. 2 §7); repeat until empty.
    4. Guard with a max-ring cap.
  • Generation: the ring set comes straight from the inward-offset primitive (Ch. 2 §7). Use the distance-field / iso-contour method (Method B) as the default: it produces the concentric rings for any topology - holes, concavity and multiple nests - in one shot (each iso-level yields every ring at that depth), which is precisely what a naive per-vertex inset cannot do. A boolean-offset library (Method A) is an exact alternative when present. This robustness is what lets the selector (§4) actually use Contour / Spiral on real artwork instead of silently degrading to a hatch look.
Edge-coverage caveat - the thin unfilled rim

Because the first ring is inset by half a spacing, the outermost drawn line sits ~½·spacing inside the true boundary, leaving a thin unfilled rim (negligible at dense shades, visible at sparse ones). When a filled-no-stroke shape must have its edge fully covered, start the first ring at ½·lineWidth inside instead, so the outer ring's outer edge lands on the boundary. (This is why "the fill reaches the boundary" is not a safe reason to suppress an outline - see Ch. 4 §2.)

  • Connectivity: one lift per ring (rings are separate), but the rings hug the shape so travel between them is tiny.
  • Use: shapes with holes, strong concavity, or hairline thickness (a few lines wide) where following the outline reproduces the form better than straight hatching. Also the natural choice for text/logos where "echo the outline" looks intentional.
  • Keyword: inward polygon offsetting / contour-parallel machining.

3.5 Spiral (connected contours) - connected ★ Auto candidate

  • Idea: Contour, but connect the concentric rings into as few continuous strokes as possible - one per nested region - by re-rooting each ring to start nearest where the tool currently is.
  • Algorithm (a greedy nearest-ring walk - the ordering logic of Ch. 7 §2-3 applied to the ring set):
    1. Generate contour rings (as §3.4), each as an open ring (drop the closing duplicate). Start the first stroke at the outermost ring.
    2. From the current stroke's end, find the nearest remaining ring and the nearest vertex on it; rotate that ring to start at that vertex and append it (tracing the full ring back to its entry). Re-rooting keeps the burned step between consecutive rings as short as one spacing.
    3. If that nearest jump exceeds a few spacings (≈3·s), the remaining rings belong to a separate nest (split off by a hole or a pinch): flush the current stroke and start a new one from the nearest remaining ring.
    4. Cap the per-ring nearest-vertex scan on very large rings (sample every k-th vertex) so the connector search stays cheap.
  • Connectivity: exactly one lift per nested region - the lift floor (§1).
  • Use: the broad default for filled regions with a real interior - round/convex blobs (zero reversals, it just spirals inward), and concave shapes, holed shapes and glyph interiors, now that the offset engine (Ch. 2 §7, Method B) produces valid rings for those too. The connectors between adjacent rings stay within the material band, so they add negligible burned length and never bridge a hole (the far-jump flush guarantees it).
  • Fallback: if ring generation degenerates (region too thin for any ring), fall back to Contour, then to the scanline fill.

3.6 Hilbert & Peano (space-filling curves) - decorative, manual only

  • Idea: clip a space-filling curve to the region; the curve visits every cell of a grid once, giving a uniform, isotropic, single-stroke fill with a distinctive woven texture.
  • Hilbert: generate via the classic d → (x,y) bit-manipulation mapping; choose the curve order so cell size ≈ spacing (order s.t. side / 2^order ≈ s), then clip the polyline to the region (Ch. 2 §6).
  • Peano: generate via its L-system (X → XFYFX-F-YFXFY+F+XFYFX, etc.; F = step, +/− = 90° turn), walk the string to a polyline, scale to the bounding box, clip.
  • Connectivity: the curve is one stroke but clipping fragments it at holes; still few lifts. Cap the order to bound the point count.
  • Use: aesthetic/texture only; never auto-selected (a connected line fill is faster for plain coverage).
  • Keywords: Hilbert curve, Peano curve, L-system, space-filling curve.

3.7 Voronoi (cellular) - decorative, manual only

  • Idea: scatter sites on a jittered grid, build each site's Voronoi cell, clip the cells to the region, draw the cell boundaries → an organic crackle pattern.
  • Algorithm: for each site, start with a large square and clip it by the perpendicular bisector half-plane against every other site (Sutherland-Hodgman, Ch. 2 §6); intersect the resulting cell with the region.
  • Determinism: derive the jitter from a hash of the grid indices so output is reproducible (no RNG). Cap the site count (cell clipping is O(sites²)).
  • Use: decorative only; never auto-selected.
  • Keywords: Voronoi diagram, perpendicular bisector, half-plane clipping.

4. The automatic selector ("Auto")

Given a region, classify it from its metrics (Ch. 2 §5) and return the time-optimal connected strategy. Auto never returns Hatch/CrossHatch/Hilbert/Peano/Voronoi - those trade time for a look and are reachable only by explicit override.

Inputs: area, elongation, rectangularity, convexity, compactness, widthInLines (= meanWidth/spacing), holes, and spacing.

Decision rules (first match wins)

The selector is tuned to be lift-optimal (§1): with a topology-robust offset (Ch. 2 §7, Method B) and a nest-aware Spiral connector (§3.5), Spiral reaches the lift floor on essentially any region with a real interior, so it is the default. Zigzag is kept where it is genuinely as good and more natural (rectangles / straight bars, where boustrophedon along the long axis gives the longest runs and ~0 lifts); Contour is kept for hairlines (nothing to spiral).

1. area < 4·spacing²                                  → ZIGZAG   # tiny: 1-2 passes, any fill ≈ same
2. widthInLines < 1.8                                  → CONTOUR  # hairline: no interior, hug the outline
3. widthInLines < 6  AND elongation > 3.5:                        # thin elongated stroke
        rectangularity > 0.88 → ZIGZAG                           #   straight bar: hatch along long axis
        else                  → SPIRAL                           #   curved/bent sliver: follow + connect
4. rectangularity > 0.82 AND convexity > 0.88 AND holes == 0
                                                       → ZIGZAG   # square / rectangle: boustrophedon
5. otherwise                                           → SPIRAL   # blobs, concave, holed, text: fewest lifts

Rationale per rule

RuleWhy
1Below a few passes the strategy is irrelevant; Zigzag is the simplest correct choice.
2A region only ~1 line wide has no interior to fill; a single contour is the fill, and there is nothing for a spiral to wind through.
3A long thin shape: a straight bar (high rectangularity) hatches along its length with the longest possible runs → Zigzag; a curved/bent sliver is followed better - and connected with fewer lifts - by a Spiral of its offset.
4A near-rectangle fills fastest with boustrophedon along the long axis: ~0 lifts and the fewest reversals (2 per scanline). A square spiral would turn 4× per ring for no gain, so squares/rectangles are pinned to Zigzag. convexity > 0.88 AND holes == 0 keeps genuinely rectangular shapes here and lets concave/holed rectangles-with-bites fall through to Spiral.
5The lift-optimal default. Round blobs (zero-reversal inward spiral), concave shapes, holed shapes and glyph interiors all connect into one stroke per nest (§3.5) - fewer lifts than Contour (one per ring) and far fewer than Zigzag fragmenting across the concavity/hole. Requires the robust offset engine; if it cannot produce rings the fill falls back (Contour → scanline, §6).

Design evolution - why this supersedes the earlier rule set

An earlier version of this selector routed holed / strongly-concave shapes to Contour and reserved Spiral for round convex blobs only. That was correct for a convex-only Spiral - one built on a naive per-vertex inset that self-intersects on concavity and cannot connect across holes. Once the offset engine is topology-robust (Method B) and the Spiral connector is nest-aware (§3.5), that restriction disappears: Spiral produces valid rings and connects them with one lift per nest on any topology, strictly beating Contour on the very lift metric the cost model (§1) is built to minimise. The lesson worth recording: an auto-selector's rules encode the current capability of the strategies behind them - strengthen a strategy (here, make the offset robust) and the selector must be re-tuned to exploit it, or it leaves the win on the table. Measured effect: a donut fill dropped from dozens of lifts (Contour) to ~2 (Spiral).

Design note

The thresholds are interpretable, ordered, and independent of any dialect. Re-tune them by changing the rationale (e.g. a machine with a very cheap lift can relax the preference for connected fills), not by trial and error.

5. "Architecture fill" override (shade-by-hatch)

A useful product mode: force every non-darkest level to a plain Hatch in a fixed user direction, while the darkest level keeps the chosen (or Auto) strategy. Because spacing already encodes the shade (§2), the lighter levels still read at the correct darkness - but as consistent parallel lines, which is the conventional way to indicate materials/tones in technical/architectural drawing. This is a dispatch-level decision layered on top of the selector: it swaps the strategy and the angle set, nothing else.

6. The fill() contract & fallback

fill(region, strategy, params) -> list<chain>:
    region.fillRule = EVEN_ODD
    spacing = max(floor, params.spacing)          # never zero or negative
    metrics = computeMetrics(flatten(region), spacing)
    if strategy == AUTO: strategy = chooseAuto(metrics, spacing)

    chains = dispatch(strategy)                    # §3
    if strategy == SPIRAL and chains empty: chains = contour(region, spacing)

    # SAFETY NET: never leave a real area uncovered
    if chains empty and metrics.area > spacing²:
        chains = hatch(region, metrics.longAxis, spacing)
    return chains
Two different area thresholds - do not unify them

Note the two area thresholds are deliberately different: 4·spacing² is the "too tiny to bother choosing" cutoff in the selector (rule 1), while spacing² is the "is there any real area left to rescue?" test for the safety-net fallback. Do not unify them.

Invariants

  • Always run on the even-odd interior so holes are respected.
  • Always clamp spacing to a positive floor.
  • Always produce output for a non-trivial area (the hatch fallback).
  • Closed rings are returned closed (first == last) so the ordering stage can re-root them (Ch. 7).

7. Implementation checklist (for an AI generating a fill engine)

Checklist
  • Flatten curves with a uniform world-space tolerance (Ch. 2 §1).
  • Run the engine on the even-odd interior; compute net area & hole count via even-odd depth (Ch. 2 §4). (Overlaps that must read solid are resolved upstream - non-zero/union per Ch. 2 §3 - before the region reaches the engine.)
  • Compute hull → min-area OBB → the six metrics (Ch. 2 §5).
  • Implement scanline span extraction once; reuse it for both Hatch and Zigzag.
  • Implement inward offset once (Ch. 2 §7); reuse it for Contour and Spiral. Prefer the distance-field / iso-contour method so holes / concavity / multiple nests work; keep a boolean-offset fast-path if available. Never ship a bare per-vertex inset as the only offset.
  • Connect Spiral rings with a nest-aware nearest-ring walk (§3.5): one stroke per nest, flush on a far jump - this is where the lift savings live.
  • Implement the auto selector as the ordered rule list (§4), and re-tune it whenever a strategy's capability changes (e.g. once the offset is robust, prefer Spiral broadly). Stale rules waste a strengthened strategy.
  • Keep density and strategy as separate inputs: density feeds the selector's spacing-relative metrics, but the reverse never holds - a strategy must not bake in its own spacing (§2). Optionally derive the shade level from fill-colour luminance (§2) - darker ⇒ denser - as a toggle.
  • Provide the hatch fallback and per-strategy caps.
  • Return world-space chains; let ordering + emission handle the rest.
In summary

A line-drawing machine covers a region with chains, not by flood-filling. The cost model (§1) says the number of lifts dominates time, so the selector only chooses among the connected fills and prefers Spiral (the lift floor) wherever the offset can produce rings. Density (§2) encodes shade separately from the path shape. The rest - outline, text, pattern - all build on this same chain foundation, then get ordered and emitted uniformly.
