Chapter 6 · Pattern

Pattern, Texture & Shade Fill

Beyond a plain solid fill (Ch. 3), a shape may be filled with a decorative pattern (lines, grid, weave, brick, dots, wave, …) that should be reproduced literally on the machine, or shaded with a tone/level that maps to fill density. This chapter covers both, and how they interact with the solid-fill engine.

1. Three kinds of "fill" - pick the right path

When a shape has a fill attribute, the first step is to classify it before generating chains:

Fill kindDetectionChain source
SolidPattern is empty / "solid" sentinelFill engine - Auto strategy (Ch. 3)
Simple hatch lookPattern is a line/grid family (line-h, line-v, diagonal, grid, cross)Hatch at the pattern's angle set
Decorative texturePattern has real repeating stroke geometry (weave, brick, wave, circles…)Pattern tiling & clip (this chapter)
Why classify first

A solid fill should be drawn the fastest way (Auto), while a decorative texture must be reproduced exactly as drawn on screen - its geometry is its identity, so it cannot be substituted with a faster strategy that produces a different shape.

2. Simple line/grid patterns → hatch angle sets

The line/grid pattern family maps to a small set of hatch angles; generate one Hatch pass (Ch. 3 §3.1) per angle:

PatternAngle set (deg)
horizontal line / dash-h{0}
vertical line / dash-v{90}
forward diagonal{45}
back diagonal{-45}
grid / plus / lattice / plaid / weave{0, 90}
diagonal grid / cross{45, -45}
(anything else){45}

Then add the shape's pattern rotation (angle + patternAngle) to every angle in the set, so the whole set rotates together with the user's setting. Spacing between lines comes from the job's hatch interval (or the shade mapping, §4). These are deliberately disjoint hatch lines - the look depends on those parallel lines existing exactly as such, never rejoined.

3. Decorative texture: tile & clip

A texture is defined once as tile geometry: a set of stroke polylines inside a repeating rectangular cell (tileW × tileH, in the texture's own pixel space). To fill a shape with it, we must reproduce exactly the brush as seen on screen:

Tile

The smallest geometric unit of a texture, containing one or more stroke polylines, repeated regularly on a tileW × tileH grid to cover any area - exactly the way an editor renders a repeating brush on screen.

// Step 1: compute the tile → shape transform
scale  = texBase × patternScale / shapeScale   // match on-screen brush size
rotate = patternAngle
// (texBase is the fixed base brush scale; shapeScale removes the shape's own
//  scaling so the texture size stays stable regardless of shape size)

// Step 2: map the shape's bounding box back into tile-pixel space (inverse transform)
for i in [i0..i1], j in [j0..j1]:   // tile indices that can overlap the shape
    if tileCount is implausibly large: bailOut()   // fallback: caller reverts to a plain fill

// Step 3: for each tile (i, j), for each stroke polyline in the tile
for each stroke in tile:
    stroke' = translate(stroke, i·tileW, j·tileH)
    stroke' = applyTransform(stroke', scale, rotate)
    runs    = clipToShapeInterior(stroke')   // segment-polygon clip, see Ch.2 §6
    append(chains, runs)

Result: the texture's strokes are tiled across the shape and clipped to its exact boundary - pixel-for-pixel the same as the rendered brush, so the preview and the machine output always agree. Work in the shape's local space, then map local→world→machine at emission.

Critical detail: the scale must match the on-screen brush

The tile→shape transform must use the exact on-screen brush scale: the same texBase × patternScale / shapeScale formula. If this scale drifts, the generated machine code will not match the design - the texture will come out too big or too small compared to what the user approved on screen.

4. Shade / tone via fill density

Independently of which kind of pattern is used, a shape can also carry a shade level. As established in Ch. 3 §2:

spacing = shadeFactor(level) × lineWidth

The darker the level, the smaller the factor, the tighter the lines, and the darker the result. A representative 4-level map:

LevelshadeFactor (× line width)
Black0.8
Dark1.5
Gray2.5
Light5.0
Two independent axes, never conflate them

Pattern / strategy decides the shape of the fill lines. Shade / density decides the spacing between them. So "light gray solid via Auto" and "black solid via Auto" use the same strategy, only different spacing. "Architecture fill" (Ch. 3 §5) is a special case: it fixes the shape to a plain hatch at lighter levels, leaving density alone to carry the shade.

5. Colour / monochrome note

For single-tool machines, the output is inherently monochrome: the tool either marks or it doesn't. On-screen colour (from a layer/theme ink) is purely a display concern; the generated toolpath carries no colour. Shade is expressed by density, not by any colour value.

shade == density

Keep colour entirely outside the geometry. It only ever affects the preview rendering, never the generated chain of points.

6. Interaction with the solid-fill engine

The dispatch (Ch. 1 §4) tries the following branches in order, for any shape that has a fill:

if pattern is decorative_texture and tiling.producedStrokes():
    use pattern chains                          // §3 - tile & clip
elif pattern is solid:
    use fill-engine chains, Auto or override     // Ch.3
else:  // simple hatch look, OR a texture that tiled empty
    use hatch chains at the pattern's angle set  // §2

The middle branch is where the smart Auto fill operates; it applies only to plain solid fills. Decorative textures deliberately bypass Auto (their appearance is the whole point). The final branch also acts as the safety fallback if a texture's tiling hits its cell-count cap and produces no strokes.

7. Checklist (when generating code for pattern / shade fills)

  1. Classify the fill: solid, simple hatch look, or decorative texture.
  2. Solid → fill engine (Auto/override); density from the shade level.
  3. Simple hatch → angle set + pattern rotation; density from hatch interval/shade level.
  4. Texture → tile with the exact on-screen brush transform, clip to the shape, cap the tile count, fall back to hatch if empty.
  5. Keep pattern-shape and shade-density as two independent axes.
  6. Keep colour out of the geometry; shade == density.
In short

Classify correctly before generating chains, honour the original geometry of a decorative texture, and keep the "shape" axis separate from the "density" axis - these three principles are enough to make every pattern and shade level come out of the machine exactly as designed on screen.
