Chapter 12 · Hardening

Inputs, Units & Edge Cases

Chapters 2-11 assumed clean, well-scaled geometry. A complete, production-grade generator must also handle the messy realities at the edges: artwork in the wrong units, self-intersecting imported paths, degenerate/empty shapes, out-of-bed geometry, unusual strokes, and the lifecycle of a long job that may be cancelled. This chapter consolidates those concerns so every case has a defined, no-crash behaviour.

Guiding rule

The generator must never crash and must never silently produce wrong output. On a bad input it must either normalise it, omit it gracefully, or record a human-readable note - never throw, never hang.

1. Source-unit normalisation → world mm

Artwork arrives from many sources in many units. Normalise everything to one world unit (mm) at the earliest possible point, then never think about units again until emission (Ch. 8 §4).

Source unitConvert to mm by
mm× 1
cm× 10
inch× 25.4

pixel / user-unit @ DPI× 25.4 / DPI
DPI ambiguity

Raster images and some vector formats carry their own DPI; if absent, assume a documented default value - commonly 96 DPI for screen artwork - and surface that assumption to the user, rather than silently guessing.

Keep geometry in floating-point mm throughout the pipeline; scale to device units only in the emitter, once, per point (Ch. 10 §7). Multiple conversion points scattered through the pipeline are a top source of scale bugs.

2. Coordinate origin & bed

Origin corner. World space and machine space each have a defined origin corner and axis directions. Make the mapping explicit and configurable (Ch. 1 §8); the emitter applies it last (Ch. 8 §5).

Check bounds in MACHINE space, not world space

Know the machine's work area. Check bounds in machine space - i.e. after the world→machine transform (Ch. 8 §5) - not in world space: a job that fits in world space can fall off the bed once mirrored/rotated (or a rotary-wrapped axis changes the meaning of coordinates entirely). In other words: map the artwork's bounding box through the same transform, then compare it to the bed rectangle.

Reject

Reject the job with a clear message ("artwork exceeds bed by N mm"). The safest, most explicit option.

Clamp

Crop the shape to fit the bed. Rarely desirable - it silently distorts the artwork.

Warn and proceed

Let the user decide, recording an annotation.

Pick a policy and make it explicit. Whichever policy is chosen, off-bed working moves must never be emitted silently.

Home always stays in raw machine coordinates

Return-home commands always use raw machine coordinates, never passing through the artwork's mirror/rotation, so "home" is always the true device origin, regardless of how the artwork has been transformed (Ch. 8 §5).

3. Path hygiene (normalise before geometry ops)

Even-odd fill, shoelace area, ray-casting, and offsetting all assume reasonable polylines. Clean the input first:

3.1 Coincident / duplicate points

Drop consecutive duplicate vertices (within ε) and zero-length segments. They break tangent/direction math (division by ~0) in smoothing, joining, offsetting, and normal computation.

3.2 Self-intersecting (non-simple) paths

Imported artwork (hand-drawn, traced, or malformed SVG) may cross itself. Effects and handling:

  • Fill: even-odd gives a predictable, deterministic result on self-intersecting paths (regions covered an odd number of times are filled), but whether that result is desired depends on intent - a self-crossing pentagram or a self-overlapping blob meant to read solid needs non-zero winding or a self-union instead. Pick the fill rule per Ch. 2 §3; the fill itself needs no geometric repair, only the correct rule.
  • Signed-area metrics (Ch. 2 §4) become unreliable on self-intersecting loops (positive and negative sub-areas cancel out). If this metric feeds the auto-fill selector, either (a) compute the metric on the even-odd rendered region rather than the raw loop, or (b) detect self-intersection and split the loop into simple sub-loops first.
  • Offsetting (Ch. 2 §7) can produce spurious self-overlaps on non-simple input; the stroke-and-subtract construction is fairly robust but still needs the result validated as non-empty.
Keywords

Bentley-Ottmann sweep-line - a sweep-line algorithm for detecting segment intersections. Polygon self-union - a boolean OR of a polygon with itself, used to repair the shape and produce a simple boundary.

3.3 Winding & fill rule

Apply the fill rule per Ch. 2 §3: even-odd for nested holes (winding-independent, deterministic hole detection), but honor the source's non-zero winding - or pre-union overlapping same-winding contours - where overlaps must read solid. Nesting-depth analysis (Ch. 2 §4) is winding-independent and valid in either case.

4. Degenerate & empty inputs - the no-crash contract

Enumerate every degenerate case and its required behaviour. Each row must be a graceful no-op or a bounded fallback, never a crash:

InputRequired behaviour
Empty document (no visible shapes)Emit a valid empty job: preamble + return-home + postamble, no working moves.
Shape fully outside the bedPer the §2 bed policy (reject/warn/clamp); never silently emit off-bed working moves.
Zero-area / collinear "shape"No fill (area ≈ 0); emit its outline if it has ≥ 2 distinct points, else skip.
Single-point subpathNot drawable as a line; skip (optionally emit a dot if the tool supports one).
Chain with < 2 pointsSkip (a chain needs ≥ 2 points to be a move).
Region thinner than one line spacingFill engine keeps the outline as the sole ring / hatch fallback (Ch. 3 §6); never leave empty output for a real area.
Fill spacing ≤ 0Clamp to a positive floor (Ch. 3 §6).
NaN / infinite coordinateReject the shape with a note; never emit NaN to a controller.
Enormous shape at tiny spacingCaps engage (Ch. 11); coarsen the shape + annotate rather than hang.

5. Stroke variants

Cross-reference to Ch. 4 §5b:

  • Dashed / dotted strokes → many short open chains (walk the boundary by arc length, emit dash segments, skip gap segments).
  • Variable-width strokes → either expand to a filled region and fill it (Ch. 3), or draw the centerline once if the tool's mark width already matches.

6. Compound paths & grouping affinity

Cross-reference to Ch. 7:

Affinity key

A compound path / glyph (outer boundary + holes) is multiple subpaths that should often stay contiguous in the output. Tag related chains with an affinity key and order the whole group as a unit if contiguity matters; otherwise, minimising raw travel is enough.

7. Job lifecycle & cancellation

A single, explicit lifecycle ties together the per-stage behaviour scattered across the previous chapters:

START
  ├─ normalise units → world mm (§1)
  ├─ resolve geometry, path hygiene (§3)
  ├─ write preamble (units, absolute, tool init, home)                 # doc 08
  │
  ├─ FOR each layer / tool-group (in output order):                       # doc 07 §5
  │     ├─ collect + plan chains (fill/outline/text/pattern)           # docs 03-06
  │     ├─ order chains (carry running position)                       # doc 07
  │     ├─ emit ordered chains × passes, with group settings           # doc 08
  │     ├─ [checkpoint] report progress; honour CANCEL                    # doc 11 §9
  │     └─ raster shapes (if any), with the same cancel checkpoints    # doc 09
  │
  ├─ write postamble (disengage tool, return home, end)                # doc 08
FINALISE
Safe postamble even on cancel

Cancellation is checked at the same cadence as progress reporting (per shape, per N raster rows). On cancel, a safe postamble (disengage tool + return home) must still be written so the machine is left in a known state - never aborting mid-working-move.

Progress is reported coarsely so the UI stays responsive (Ch. 11 §9). Incremental regeneration: only re-plan shapes/layers that changed; cache the rest (Ch. 11 §9).

8. Checklist (hardening a generator)

  • Normalise all source units to world mm at import; a single conversion point.
  • Make the origin/bed mapping explicit; define a bed-bounds policy; keep return-home moves in raw machine coordinates.
  • Clean paths: drop duplicate points / zero-length segments; apply the correct fill rule per Ch. 2 §3 (even-odd for nesting; non-zero/union for overlaps).
  • Handle self-intersection (even-odd for fill; metrics on the rendered region, or split into simple loops).
  • Fully implement the degenerate-input table (§4) - every row a graceful no-op or bounded fallback; never crash, never emit NaN.
  • Support dashed & variable-width strokes.
  • Optional: compound-path affinity in ordering.
  • Implement the job lifecycle with cancel checkpoints and a safe postamble on cancel.
Closing the book

From the chain model in Chapter 0, through architecture, geometry, fill algorithms, outlines, single-stroke text, patterns, ordering, machine-code emission, raster, curve fidelity, and performance - this chapter closes the loop by hardening the system against the messy reality of real data. A good generator is not one that has never met a bad input, but one that already has an answer for every bad input.
