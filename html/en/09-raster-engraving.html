Chapter 9 · Raster

Raster Engraving

A bitmap image is not a chain source. You cannot meaningfully "outline" or "fill" a photograph with polylines; instead the tool sweeps across the image row by row and modulates power per pixel (dark = strong, light = off). This approach only applies to machines that can vary power while moving (laser, variable-pressure engraver) - a pen or blade normally cannot render tone, so raster shapes are simply skipped.

This is the one part of the system that bypasses the chain/ordering pipeline learned in earlier chapters; it emits its own moves directly. The chapter still belongs in the book because "convert artwork to machine code" must handle images too, not just vector shapes.

Skipped for non-modulating tools

If the selected tool is a pen or blade (unable to vary intensity while moving), the raster stage is skipped entirely - it does not try to "fake" tone with stroke density, since that is the job of pattern/hatch fill (Chapter 6), not this raster engraver.

1. The model

The whole path from image to machine code fits into four consecutive transforms: sample the image onto a physical grid, turn the grayscale values into a power grid, then scan that grid boustrophedon-style and merge cells of equal power into a single stroke.

Image (pixels + tone)→ mm grid at the scan step→ Power grid→ Boustrophedon scan, run merge→ Travel / Working move

2. Sampling grid: never oversample

The scan step is chosen from both the desired engraving spacing and the image's source resolution:

colStep = max(interval, imageWidthMm  / imagePixelsWide)
rowStep = max(interval, imageHeightMm / imagePixelsTall)
  • interval - the spacing between engraving lines (material/beam dependent).
  • max(...) prevents oversampling the bitmap: scanning finer than one source pixel is pointless, it only adds motion without adding information.
  • Bounded work: cap the total cell count (rows × cols ≤ kMaxCells). If exceeded, coarsen both steps by a factor of √(cells / kMaxCells) and record a human-readable note. This stops a large image at a tiny scan interval from producing millions of moves.

Sample each cell at its center, in the image's own local space, so the scan rides the image's own transform (rotation/scale). Cells outside an optional clip mask are treated as white (skipped, not engraved).

3. Tone reproduction: two modes

There are two ways to turn each cell's gray value into engraving power: continuous modulation through a gamma curve, or quantisation to 1-bit via dithering.

ModePower outputSuited when
Grayscale + gammaContinuous, within [minPower, maxPower]Smoothly modulating tool (diode laser), image with a wide gray range
1-bit dithering (Floyd–Steinberg)Binary, 0 or full powerOn/off-only tool, or a crisper "dot" look is preferred over a soft gray field

3.1 Grayscale power modulation (with gamma)

Map cell tone directly to a power level within [minPower, maxPower]:

v = 255 - gray            # invert: darker burns hottest
if v <= 0: power = 0      # white → off (hard skip, no engrave)
t = v / 255
if gamma != 1: t = t^gamma   # tone curve: correct for the material's non-linear response
power = minPower + round(t × (maxPower - minPower))
  • minPower - a power floor so light gray areas still leave a mark (below the tool's activation threshold, nothing shows up at all).
  • gamma - bends the tone curve to match how the material darkens with actual power (non-linear).
Terminology

Power/PWM modulation - modulating output power (usually via PWM) to the desired value at each point. Gamma correction - the power-law transform t^gamma that compensates for the non-linear response between control signal and observed darkness on the material.

3.2 Dithering (1-bit) - Floyd–Steinberg

For an on/off-only tool, or for a crisper look, convert to on/off dots that approximate tone through their density. Floyd–Steinberg error diffusion: threshold each cell to black/white, then push the quantisation error to the not-yet-visited neighbouring cells using the 7/16, 3/16, 5/16, 1/16 kernel:

for each cell (row-major):
    old = buf[cell]; new = (old < 128) ? 0 : 255
    err = old - new; buf[cell] = new
    buf[right]       += err × 7/16
    buf[below-left]  += err × 3/16
    buf[below]       += err × 5/16
    buf[below-right] += err × 1/16

Dithered cells are already 0/255, so they engrave at full power (no gamma applied). Other diffusion kernels (Atkinson, Jarvis) or ordered/Bayer dithering are alternative choices.

Terminology

Error diffusion - the technique of spreading quantisation error to nearby unprocessed samples so the local average tone is preserved. Halftoning - reproducing continuous tone through the density of binary dots, a concept originating in printing.

4. Scanning: boustrophedon and run merge

leftToRight = true
for each row r:
    compute power for every cell in the row
    find firstBurn..lastBurn (# skip a fully-white row, still flip direction)
    walk that span in the current direction:
        merge a maximal RUN of cells with equal power
        power > 0 → engaged working move to the run's end, at that power
        power = 0 → travel move over the gap (white/clipped cells)
    leftToRight = !leftToRight       # flip direction each row
  • Boustrophedon - "ox-plough" back-and-forth rows, avoiding a return trip to the margin on every scan line.
  • Run-length merge - collapses adjacent cells of equal power into one move, dramatically reducing the line count. Combined with modal power (Chapter 8 §3), a smooth gradient becomes a handful of moves per run instead of one per pixel.
  • Gaps (white or clipped) are traversed while disengaged (rapid), so masked regions are skipped rather than mistakenly engraved.
Why run merge matters

Without merging, a 1000×1000-pixel image could produce a million individual moves. Run-length merge turns a smooth gradient into a small number of constant-power segments - the move count scales with the number of power changes per row, not with the pixel count.

5. Dynamic power (M4) vs constant power (M3)

For raster, prefer the controller's dynamic power mode (GRBL M4) over constant (M3): it scales beam power with the actual feed rate, keeping engraving density even through the acceleration/deceleration segments at each row's ends. Vector shapes typically use constant power (M3).

M4 - dynamic power
Power scales with the instantaneous feed rate. Used for raster: the slower accel/decel segments at the start/end of a scan row still yield even engraving density, without "over-burning" at the ends.
M3 - constant power
Power stays fixed regardless of speed. Used for vector (chains): speed along a cut/draw path is usually controlled separately, so power doesn't need to compensate for feed rate.
Track modal state

Just like power and feed in Chapter 8, the M3/M4 mode is modal state: emit the mode-switch command only once when transitioning from a vector segment to a raster segment (or back), not on every scan row.

6. Integration with the vector pipeline

Raster bypasses chains and ordering, but it still lives inside the same job - don't let this bypass drop it out of the shared machinery:

Coordinate transform

Raster points go through the same world→machine format() transform as chains (Chapter 8 §5): sample in the image's local space, map local→world, then let the emitter apply the machine affine (home/mirror/rotate/rotary). Skipping this step lands the engraving in the wrong place or orientation.

Per-layer sequencing

Within a layer, emit the ordered vector chains first, then that layer's raster shapes (Chapter 12 §7); carry the running position so the first raster row starts near where the vectors just ended.

Passes

Raster participates in the layer's pass count just like chains: re-emit the whole raster scan once per pass.

Settings

Raster uses the layer's power as the max/on level and the layer's feed rate; min-power/gamma/dither are the layer's own image controls (§3).

Common mistake

Because raster "emits its own code," it's easy to accidentally miss the machine transform (home/mirror/rotary) that vector chains always pass through in Chapter 8. Reuse a single format() function for both paths - don't write a separate copy for raster.

7. Not animated in preview

Raster produces an enormous number of moves; don't feed them into the toolpath simulator. Instead, show raster as a static image overlay on the preview. The vector simulator then only animates chains, keeping the interface responsive (raster moves are emitted with no simulator index assigned).

General principle

The preview needs to be honest about output shape, not necessarily simulate every single motion. With tens of thousands of raster moves, a static overlay image conveys the right information at nearly zero cost.

8. Checklist (AI generating a raster engraver)

  • Skip raster entirely for non-modulating tools (pen/blade).
  • Scan step = max(interval, one source pixel); cap total cell count and coarsen if needed (with a note).
  • Sample cell centers in the image's local space; honour the clip mask.
  • Tone: grayscale+gamma into [minPower, maxPower], OR Floyd–Steinberg dither to 1-bit.
  • Scan boustrophedon; merge equal-power runs; working move over engraved spans, travel over gaps (GRBL G1/G0).
  • Use dynamic power (M4) for even tone; track modal power/feed.
  • Keep raster out of the animated simulator.
