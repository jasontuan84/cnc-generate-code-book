Chapter 0 · Foundations

The core idea: the Chain model

Before we discuss curves, fills, or text, we need one simple truth as a foundation: everything a point-to-point machine can do reduces to a single data structure - a chain of points. Grasp this, and the whole system becomes clear.

1. Every machine speaks the same "abstract language"

A pen plotter, a laser, a vinyl/knife cutter, a CNC router - however different their physics - all accept the same basic instruction set:

Move while the tool is DISENGAGED

Pen up / laser off / blade up. This is a travel (rapid) move: it produces nothing on the material.

Move while the tool is ENGAGED

Pen down / laser on / blade down. This is a working move: it produces a mark, a cut, or a burn.

Modulate the tool (optional)

During a working move, the intensity can vary: laser power, spindle RPM, blade force.

2. Defining the "Chain"

Because there are only two kinds of motion, every drawing reduces to a single data structure:

Chain

A chain = an ordered list of 2-D points (a polyline) that the tool traces in one continuous engaged pass. Between chains, the tool disengages and travels.

The entire job fits in one sentence: produce a set of chains that reproduce the artwork, order them to minimise travel, then serialise them into the target machine language. This is the single most important abstraction in the whole system.

The core separation principle

Keep the geometry (chain production) completely independent of the machine language (chain serialisation). Only the final emission stage - thin and unique - knows about a specific machine. If a function computing a hatch pattern mentions G1 or PD, the design is wrong.

3. The five-stage pipeline

Turning a drawing into code passes through five distinct stages. Each has a single responsibility and communicates with the next through a clear contract.

1 · Resolve geometry
Bake every transform (translate/rotate/scale/shear) into world space; choose the fill rule; identify holes. → Ch. 2
2 · Extract primitives
Flatten Bézier/arc curves to polylines at a controlled tolerance; separate the outline from the interior area. → Ch. 2, Ch. 4
3 · Plan chains
Per source: solid area → fill engine; pattern → tile & clip; text → skeleton; image → scanline modulation. → Ch. 3, 5, 6, 9
4 · Order chains
Minimise travel with nearest-neighbour, choose entry points, re-root closed loops. → Ch. 7
5 · Emit dialect
Modal tool state, coordinate formatting, preamble/postamble, coordinate-system transforms. → Ch. 8

These terms recur throughout the book. You need not memorise them now - return here when you hit an unfamiliar word.

Pen-up / pen-downGeneric names for disengage / engage, whatever the tool.
FlatteningApproximating curves (Bézier/arc) by short line segments.

HatchParallel straight fill lines.
Boustrophedon"Ox-plough" back-and-forth path; alternating scanline direction.
Contour / offset fillConcentric inward copies of the boundary.
Kerf / tool offsetPhysical width the tool removes/leaves; requires geometric offsetting.
OvercutA small extension past a closed loop's start so a blade fully severs it.
Modal stateA machine setting that persists until changed (units, power, feed).
DialectA concrete command language (GRBL, HPGL, Marlin…).
Distance fieldA grid where each interior cell holds its distance to the boundary; its iso-lines are the inward offsets (used for robust Contour/Spiral, Ch. 2).
Iso-contourA curve of constant value in a field; here, the set {distance = d} = the inward offset by d. Extracted with marching squares.
NestA maximal group of concentric offset rings reachable without a lift; a hole or pinch starts a new nest. Spiral emits one stroke per nest.
LiftA pen-up / disengage-travel-re-engage cycle; the dominant time cost, hence fills are optimised to minimise it (Ch. 3).
LuminancePerceptual brightness 0.299R+0.587G+0.114B of a fill colour; drives shade-by-colour fill density (Ch. 3).

5. Seven principles for generating this code

  1. Never couple geometry to a dialect. Geometry returns chains; a separate emitter turns chains into text.
  2. A chain is the contract. Design each feature to append chains to a list. Ordering and emission then work uniformly for all of them.
  3. Every tunable has a rationale. Spacing, tolerance, thresholds - state why the value is what it is, so it can be re-tuned intelligently.
  4. Prefer connected paths over disjoint ones. Time cost is dominated by tool lifts and travel, not by engaged distance (see the cost model in Ch. 3).
  5. Always have a fallback. Any strategy can degenerate on a pathological shape; a robust generator falls back to the simplest correct method rather than emitting nothing.
  6. Prefer topology-robust primitives. A primitive that quietly fails on holes or concavity (e.g. a naive per-vertex inset) makes the selector above it lie: it picks the right strategy, which then silently degrades. Build the offset on a distance field (Ch. 2) so it works for any topology, and the whole fill stack becomes honest.
  7. An auto-selector encodes its strategies' current capability. When you strengthen a strategy (make Spiral topology-robust, see Ch. 3), re-tune the selector to exploit it - otherwise the improvement is left unused. Selector rules are not permanent; they track what the strategies behind them can now do.
Grasp this chapter and you have half the picture

The rest of the book simply answers: how to produce the best chains for each source type (fill, outline, text, image), how to order them, and how to translate them into code. It all rests on the chain model you just learned.
