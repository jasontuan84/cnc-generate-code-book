Chapter 11 · Performance

Performance & Complexity

Good output is not enough - generation must also stay responsive. A user editing artwork expects toolpath regeneration in interactive time, and heavy jobs (fine fills, large images, thousands of glyphs) must never be allowed to hang the app or exhaust memory. This chapter lists the complexity of every stage, the hard caps that keep them bounded, and the techniques that make them fast.

Golden rule

Every stage must have a known worst-case bound. An unbounded loop that runs on "spacing over area", or an O(n²) pass with no cutoff, will sooner or later meet an input that freezes the app.

2. Flattening: fidelity vs. point count

Point count is set by tolerance (see Chapter 10 §2). More points ripple downstream through every later stage (clipping, ordering, emission, streaming). Therefore:

  • Use adaptive subdivision so straight spans cost almost no points, and only tight curves receive extra ones.
  • Never set tolerance below the tool's own resolution - extra points buy no quality but cost throughout the whole pipeline.
  • Flatten once per shape and cache the result if the shape is unchanged.

3. Fills

Scanline (hatch / zigzag)

For every scanline we test crossings against every boundary edge → O(scanlines × edges). Scanline count equals height / spacing, so a very small spacing on a tall shape explodes. Guards:

  • A spacing floor (never below a sane minimum) caps the scanline count.
  • The crossing test is trivial arithmetic; this is the cheapest fill.
  • Optional: bucket edges by y-range so each scanline only tests edges that can cross it (turns the edge factor into a near-constant).

Offset (contour / spiral)

There are two different cost profiles, one per offset method (see Chapter 2 §7):

Method A - boolean erosion

Each inward offset pass costs one outline stroke plus one boolean subtract - the most expensive fill. Ring count = meanWidth / spacing.

Method B - distance field + iso-contours

One rasterise pass plus one distance transform (O(cells)), then a single marching-squares pass (O(cells + total contour length)) yields all rings at once.

Method A needs a max-ring cap (e.g. a few thousand) so that a tiny spacing on a huge area still terminates; and it stops as soon as an inset pass returns empty.

Method B's cost is set by the grid, not by the ring count - so it does not blow up as spacing shrinks (a smaller spacing just reads more iso-levels off the same field). It needs a max-cell cap (e.g. ~500k for a pure-language pass): if (w+pad)(h+pad)/h² > cap, enlarge the grid step h by √(area/cap) and record it; the default grid step is h ≈ spacing/3. A chamfer transform is two linear scan passes; marching squares emits crossings in a single pass over cells (only boundary cells produce segments) - both run comfortably under a second within the cell cap.

For both methods: prefer scanline fills for large plain areas; reserve contour/spiral for shapes whose own shape calls for them (the auto selector already handles this, see Chapter 3 §4).

Decorative (space-filling curves, Voronoi)

  • Hilbert/Peano: cap the curve's order (e.g. 2⁷ or 3⁴ cells/side) so the point count cannot blow up.
  • Voronoi: naive cell clipping is O(sites²); cap the site count (e.g. ≤ 240). Use a spatial structure (or a real Fortune's algorithm Voronoi) if more cells are needed.

4. Clipping: the hidden quadratic

Clipping a polyline against a general region tests every segment against every boundary edge → O(seg × boundaryEdges). Pattern tiling multiplies this by the tile count. This is the most common accidental hotspot. Mitigations:

A commonly missed hotspot

A pattern-tiling loop (see Chapter 6) looks harmless but multiplies segment count by tile count and by boundary edge count - easily turning a seemingly cheap operation into a quadratic pass that runs for seconds.

  • Bounding-box reject: skip any (segment, edge) pair whose bounding boxes don't overlap - turns most comparisons into a cheap, early-out test.
  • Spatial index on boundary edges (uniform grid, quadtree, or an interval tree on y). Query only edges near the segment being tested. Brings cost down to near O(seg · log).
  • Tile-count guard: if the estimated tile count is implausibly large, fall back to a plain fill rather than tiling millions of cells.

5. Ordering: O(n²) with a cutoff

Greedy nearest-neighbour (see Chapter 7) is O(n²). Dense fills produce tens of thousands of chains, so:

  • A cutoff (e.g. 4000 chains): above it, skip the O(n²) pass and use the orientation-only O(n) fallback, which trusts the fill's inherent locality (scanline/ring order).
  • For large-but-important sets, use a spatial sort (grid order or a Hilbert order on chain midpoints) → O(n log n), giving a much better tour than raw source order.
  • Use squared distances (no sqrt) in the inner loop.

6. Raster: cap and coarsen

Raster is O(rows × cols) and can reach millions of cells (see Chapter 9 §2):

  • Step ≥ one source pixel (never oversample) and ≥ the interval.
  • Max-cell cap: if rows × cols > kMaxCells, coarsen both steps by √(cells/cap) and record a note.
  • Run-length merging plus modal power collapsing compress a gradient into far fewer moves.
  • Don't feed raster moves into the animated simulator (preview as a static image instead) - animating millions of moves would kill UI responsiveness.

7. Skeletonisation: bound the bitmap

Single-stroke text centerlines (see Chapter 5) rasterise then thin iteratively:

  • Cap resolution and total pixel count (e.g. max dimension ~2600 px, ≤ ~9M px total).
  • Thinning is O(pixels) per iteration, with a bounded (guarded) iteration count.
  • Skeletonise per glyph, not per whole text block, so every bitmap stays small.

8. Emission & streaming

  • Trim coordinate precision and strip trailing zeros (see Chapter 8 §4): fewer bytes ⇒ deeper controller look-ahead ⇒ smoother dense motion, and less string-building work.
  • Modal-state suppression (see Chapter 8 §3) removes redundant lines - smaller output, faster to build and send.
  • Build output with a pre-sized buffer / streaming writer, not repeated string concatenation (avoid O(n²) string growth).
  • For very large jobs, stream to the device/file incrementally rather than materialising the whole program in memory.

9. Keeping the UI responsive

  • Report progress at a coarse cadence (e.g. once per shape, or every N raster rows) so a long job shows a moving progress bar, not a frozen window.
  • Generate off the UI thread (worker/background) for heavy jobs; post results back afterwards. Keep the geometry pure/thread-safe so this stays simple.
  • Cancellation: allow a long generation job to be cancelled; check a flag at the same cadence as progress.
  • Incremental regeneration: only regenerate shapes/layers that changed; cache the rest. Flattening and filling an untouched shape should never be repeated.

10. Determinism - a rule tied to performance

Controlled randomness

Avoid RNG in geometry (e.g. Voronoi jitter): derive randomness from an index-based hash so output is reproducible across runs. Deterministic output makes caching valid and debugging tractable; a nondeterministic generator cannot safely skip "unchanged" work.

11. Checklist (AI keeping generation fast)

  • Every iterative construction has a hard cap (rings, order, sites, cells, iterations).
  • Adaptive flattening; never refine below tool resolution; cache per shape.
  • Clipping uses bbox reject plus a spatial index; guard tile counts.
  • Ordering switches to O(n) / spatial sort above a chain-count cutoff; squared distances only.
  • Raster caps and coarsens; run-length merges; never fed to the animated simulator.
  • Emission trims precision, suppresses modal repeats, streams large jobs.
  • Heavy jobs run off the UI thread with progress and cancellation; regenerate incrementally.
  • Geometry is deterministic (hash-based, not RNG).
